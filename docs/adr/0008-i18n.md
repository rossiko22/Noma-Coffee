# ADR 0008 — Internationalization: bilingual EN/DA

**Status:** Proposed · **Date:** 2026-06-23

## Context
Decision: **bilingual EN + DA**. The café is in Copenhagen; Danish content strengthens local-SEO reach to
locals while English serves expats/tourists. The current EN/DA toggle is cosmetic and does nothing.

## Decision
- **Localized routes:** English at the root (`/...`), Danish under a **`/da` prefix** (`/da/cafe`, …). Each
  locale is fully prerendered.
- **Content:** translated in **Sanity** via document/field internationalization ([0002](0002-headless-cms.md));
  UI strings via a lightweight message catalog.
- **SEO:** emit **`hreflang`** alternates (incl. `x-default`), per-locale **canonicals**, and **per-locale
  entries in `sitemap.xml`**. Localize `<html lang>` and OG locale.
- Replace the cosmetic toggle with a real language switcher that preserves the current page.

Rejected: query-param or cookie-based locale (worse for SEO/crawlability); separate domain per language
(unneeded complexity at this scale); auto-translating without editorial control (brand voice matters).

## Consequences
- Two prerendered variants of each page — modest build-time increase, no runtime cost.
- The client must maintain content in both languages (acceptable; Studio makes this explicit). A locale may
  fall back to EN where DA is absent, documented for editors.
- Clean, crawlable bilingual structure that search engines index correctly.
